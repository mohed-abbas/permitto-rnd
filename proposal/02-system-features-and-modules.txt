================================================================================

================================================================================
2.1 USER MANAGEMENT & AUTHENTICATION MODULE
================================================================================

  Feature                  | Description
  -------------------------|-----------------------------------------------------
  User Registration        | Admin-invited registration with email activation
  Login / Logout           | Email/password authentication with secure session
                           | management
  Password Reset           | Self-service password reset via email link
  Session Management       | JWT access tokens (short-lived) + refresh tokens
                           | (long-lived) with automatic renewal
  Role Assignment          | 5 roles: Super Admin, PTW-PC (Coordinator),
                           | WPI (Issuer), HSSE Officer, WPR (Receiver)
  User Profiles            | View/edit personal details, notification
                           | preferences, and one-time digital signature setup
  Digital Signature Setup  | User draws and saves their signature once during
                           | profile setup. This signature is stored securely in
                           | the database and reused for all approvals - no need
                           | to redraw each time
  User Deactivation        | Soft-delete - deactivated users cannot login but
                           | historical records are preserved


================================================================================
2.2 ROLE-BASED ACCESS CONTROL (RBAC) MODULE
================================================================================

  Role              | Capabilities
  ------------------|------------------------------------------------------------
  Super Admin       | Create projects, manage all users (all roles), system-wide
                    | visibility, system configuration
  PTW-PC            | Create permit templates, create Issuers & Receivers,
  (Coordinator)     | 3rd-level approval, close-out permits, manage PTW Register
  WPI (Issuer)      | 1st-level approval, create Receivers, daily re-approval,
                    | SIMOPs countersign, grant/deny permit extensions
  HSSE Officer      | 4th-level (final) approval, independent risk assessment
                    | verification, request additional safety measures, HSSE
                    | compliance oversight
  WPR (Receiver)    | Request permits, fill multi-step forms, upload documents,
                    | daily re-confirmation, request permit extensions

  Permissions enforced at both API level (backend middleware) and UI level
  (feature visibility).

2.7 4-LEVEL APPROVAL WORKFLOW MODULE
================================================================================

  Approval Level | Actor             | Actions
  ---------------|-------------------|------------------------------------------
  Level 1        | WPI (Issuer)      | Review permit, verify risk assessment,
                 |                   | approve / request additional measures /
                 |                   | reject
  Level 2        | PTW-PC            | Review permit, approve / reject
                 | (Coordinator)     |
  Level 3        | HSSE Officer      | Independent risk assessment verification,
                 |                   | approve / request additional measures /
                 |                   | reject

  Feature                              | Description
  -------------------------------------|----------------------------------------
  One-Click Approval with Stored       | On approval, user clicks "Approve" and
  Signature                            | their stored profile signature is
                                       | automatically applied with a fresh
                                       | timestamp. No need to redraw the
                                       | signature each time - the authenticated
                                       | click is the legal authorization
  Request Additional Measures          | Approvers can request specific
                                       | additional safety measures from WPR -
                                       | not just flat reject. WPR implements
                                       | measures and resubmits
  Rejection with Feedback              | Mandatory reason/comments on rejection
                                       | - WPR receives feedback and can revise
                                       | & resubmit
  Approval Dashboard                   | Each role sees their pending approvals
                                       | with filters by project, zone, type,
                                       | urgency
  Approval Notifications               | Real-time notifications at each workflow
                                       | transition to all relevant parties
  Audit Logging                        | Every approval, rejection, and
                                       | additional measures request logged with
                                       | timestamp, user, and comments

2.8 SIMOPs MANAGEMENT MODULE
================================================================================

  Feature                  | Description
  -------------------------|-----------------------------------------------------
  Automatic Detection      | System detects when multiple permits are active or
                           | pending in the same zone during overlapping time
                           | periods
  SIMOPs Gate              | Blocking step in approval flow - if SIMOPs detected,

                           | documenting how interfaces between concurrent
                           | operations are managed
  Cross-Permit Visibility  | All parties can see which other permits are active
                           | in the same zone

2.15 FILE STORAGE MODULE
================================================================================

  All file storage is self-contained on the client-provided VPS - no external
  cloud storage services required.

  Feature              | Description
  ---------------------|--------------------------------------------------------
  Server-Side Storage  | Supporting documents stored securely on the VPS
                       | filesystem in access-controlled directories
  Secure Upload        | Files uploaded through the API with authentication and
                       | authorization checks
  Supported Formats    | PDF, JPG, PNG, HEIC
  Size Limits          | 25MB per file, 10 files per permit
  Access Control       | File access respects permit visibility permissions -
                       | only authorized users can download
  Signature Storage    | User signatures stored as encrypted data in the
                       | database (PostgreSQL) - not as separate files
  PDF Storage          | Not applicable - PDFs are generated on-demand and
                       | cached client-side by the PWA

  STORAGE ARCHITECTURE SUMMARY:

  Data Type              | Storage Location        | Notes
  -----------------------|-------------------------|----------------------------
  User signatures        | PostgreSQL database     | One-time profile signature,
                         |                         | stored securely, reused on
                         |                         | all approvals
  Permit data            | PostgreSQL database     | All form data, status,
                         |                         | audit logs, timestamps
  Supporting documents   | VPS filesystem          | Uploaded files stored in
                         |                         | secure server directories
  PDFs                   | Not stored              | Generated on-demand, cached
                         |                         | by PWA for offline
  Offline form drafts    | Browser (IndexedDB)     | Encrypted, synced to server
                         |                         | on reconnect

  No external storage services (Cloudflare R2, AWS S3, etc.) are required.
  The entire system runs self-contained on the client-provided VPS.

2.18 PROGRESSIVE WEB APP (PWA) & OFFLINE MODULE
================================================================================

  OFFLINE CAPABILITIES:

  Feature                  | Description
  -------------------------|-----------------------------------------------------
  Installable              | Users can install the app on mobile/desktop home
                           | screen - works like a native app
  Offline Permit Viewing   | View active permits and permit details without
                           | internet connectivity
  Offline PDF Viewing      | PDFs for active permits are pre-cached when online
                           | and available for viewing/printing offline
  Offline Form Filling     | Fill permit forms offline - data stored locally in
                           | encrypted browser storage
  Queue Submissions        | Submissions and approvals queued when offline, with
                           | visual indicator of pending actions
  Auto-Sync                | Automatic synchronization when connectivity is
                           | restored - queued actions are submitted to the
                           | server
  Sync Status Indicator    | Visual indicator showing online/offline status and
                           | number of pending sync actions
  Smart Caching            | Only the user's own active permits and their PDFs
                           | are cached - not all system data
  Cache Refresh            | When permit data changes (extension, daily approval,
                           | status change), cache is refreshed on next online
                           | access


  OFFLINE SECURITY MEASURES:

  All offline cached data is protected with industry-standard security
  measures, following the same model used by enterprise applications
  (SAP Fiori, Microsoft Teams, SafetyCulture).

  #  | Security Measure         | Description
  ---|--------------------------|--------------------------------------------
  1  | HTTPS Only               | All data in transit is encrypted. Service
     |                          | workers only function over HTTPS - enforced
     |                          | by browsers
  2  | Encryption at Rest       | All cached data in browser storage
     |                          | (IndexedDB) is encrypted using the Web
     |                          | Crypto API with a key derived from the
     |                          | user's authenticated session. Even if
     |                          | someone accesses the device storage
     |                          | directly, the data is unreadable
  3  | Session-Bound Cache      | Cached data is tied to the authenticated
     |                          | session. When the session expires or user
     |                          | logs out, the encryption key is destroyed -
     |                          | cached data becomes permanently unreadable
  4  | Wipe on Logout           | All local data (IndexedDB, Cache API,
     |                          | service worker cache) is completely wiped
     |                          | when the user logs out
  5  | Session Timeout          | Auto-lock after configurable idle period
     |                          | (e.g., 15-30 minutes). User must
     |                          | re-authenticate to access cached data -
     |                          | even while offline
  6  | Data Minimization        | Only the user's own active permits are
     |                          | cached - not all system data.
     |                          | Closed/expired permit data is automatically
     |                          | purged
  7  | Auto-Expiry              | Cached data expires automatically after a
     |                          | configurable period (24-48 hours) regardless
     |                          | of session status
  8  | Offline Action Audit     | All actions taken offline are logged with
     |                          | timestamps. On sync, the server validates
     |                          | and records every offline action in the
     |                          | audit trail


  OPTIONAL ENHANCED SECURITY (Available on Request):

  #  | Security Measure     | Description
  ---|----------------------|------------------------------------------------
  9  | Remote Wipe          | Admin can flag a user's session for device wipe.
     |                      | On next sync, all local data is cleared - useful
     |                      | for lost or stolen devices
  10 | Biometric Lock       | Require device fingerprint or face ID before
     |                      | accessing offline cached data
